# 多曝光 HDR 合成

本文处理同一天体、同一视场的不同曝光组，例如 M42 短曝光核心与长曝光外围。它不同于管线中的 `protected_hdr_compress`：前者合并真实多曝光数据，后者只压缩单张成片的高光。

## 当前能力边界

- 当前仓库提供亮度域高光压缩，但没有原生多曝光配准与 HDR 堆栈 CLI。
- 多曝光校准、配准和线性 HDR 合成应先在 Siril/PixInsight 等外部工具完成。
- 合成结果可进入 `pipeline.py` 做后续拉伸和高光保护。

## 输入要求

- 同一设备、滤镜、构图和焦点。
- 每组曝光分别完成校准与堆栈。
- 所有曝光母版配准到同一参考图。
- 使用 32-bit 线性 FITS/TIFF。
- 至少一组短曝光核心未饱和。
- 长曝光负责外围低信号，不用于恢复已经饱和的核心。

## 曝光组设计

常见三级：

```text
short: 1-10 s      保护极亮核心和亮星
medium: 30-60 s    连接核心与主体
long: 180-600 s    获取外围暗弱结构
```

实际时间取决于设备、目标和天空背景。关键是曝光覆盖重叠，而不是固定秒数。

## 合成原则

每组先匹配背景和线性响应。以短曝光为高光来源，长曝光为暗部来源：

```text
HDR = w_short * short
    + w_medium * medium
    + w_long * long
```

权重随亮度变化：

- 高光接近饱和时增加短曝光权重。
- 中间调使用平滑过渡。
- 暗部优先长曝光。

权重蒙版必须平滑，并基于亮度而不是手绘结构。禁止把不同视场的参考图作为 HDR 层。

## 饱和检测

每组检查：

- P99/P99.9。
- `>=0.995` 的裁切比例。
- 核心区域是否形成平顶。
- 星点中心是否失去渐变。

已经饱和的像素只能由更短曝光替换，不能通过 HDR 压缩恢复。

## 合成顺序

```text
各曝光组独立校准与堆栈
→ 统一配准和裁切
→ 背景/尺度匹配
→ 构建亮度权重
→ 线性 HDR 合成
→ 检查接缝和光晕
→ 输出 float32 母版
→ 常规后期管线
```

推荐后处理：

```bash
python scripts/pipeline.py hdr_linear.tif final.jpg \
  --strength adaptive \
  --target-type emission_nebula \
  --target-name M42 \
  --steps color,stretch,enhance,sharpen,final_color,style,final_denoise \
  --keep-all
```

M42 安全规则会提高核心保护，但不能替代真实短曝光。

## 单张 HDR 压缩

对于已经合成或只有单张的数据，可使用管线的亮度域 HDR：

```json
{
  "hdr_strength": 0.35,
  "hdr_knee_percentile": 85
}
```

它只压缩 knee 以上的亮度，不恢复饱和信息。若出现暗环、亮边或核心变灰，应降低强度。

## 质量审查

- 核心到外围过渡连续，无明暗接缝。
- 星点大小不会在曝光层边界突变。
- 背景噪声统计一致。
- 核心细节来自短曝光，不是锐化伪影。
- 合成后综合色彩未改变。
- 无鬼影、双星、配准错位。

## 常见失败

- 核心仍死白：短曝光仍然饱和。
- 核心周围暗环：权重过渡太窄或局部 HDR 过强。
- 星点双影：曝光组配准失败。
- 背景断层：各组背景尺度未匹配。
- 暗部噪声过大：长曝光信噪不足，不能靠权重放大补救。
