# Starun MVP 产品与系统设计

## 1. 文档目的

本文定义 Starun 第一个可交付里程碑的产品范围、系统架构、任务契约、Agent 骨架、资源约束和验收标准。

本文是对 `STARUN-PRD.md` 的业务与技术补充。原 PRD 中的深空暗色视觉、色彩、字体、卡片和响应式原则继续有效；若两份文档冲突，以本文为准。

## 2. 产品定位

Starun 是面向有经验天文摄影爱好者的在线后期辅助平台。用户上传已校准、已叠加的单张线性 FITS 文件后，可以：

1. 获取图像基础信息、专业指标、分步骤后期方案和推荐参数。
2. 选择一种风格，让 AI Agent 无人值守地完成自动出图。
3. 在当前浏览器查看分析和处理历史。

首个里程碑用于验证产品流程、任务基础设施和 Agent 编排接口，不交付真实专业分析算法或真实自动后期能力。

## 3. 目标用户与体验原则

### 3.1 目标用户

MVP 面向熟悉天文摄影和基础后期术语的爱好者。界面可以直接使用 FITS、HDU、SNR、FWHM、拉伸、降噪、反卷积和色彩校准等术语，不提供新手与专业双模式。

### 3.2 核心原则

- 专业但不复杂，优先展示用户当前需要判断的信息。
- 自动处理完全无人值守，不在中间步骤要求确认。
- Agent 的计划、工具调用、参数、观察和结果可追踪、可复现。
- 科学真实性是所有风格的底线，不允许生成原始数据中不存在的天体结构。
- Mock 指标和演示产物必须显著标识，不得伪装为真实算法结果。
- 无账户体验优先，接受本地历史不跨设备、不保证永久保存的限制。

## 4. 产品范围

### 4.1 页面

产品包含四个一级页面：

- `/`：产品首页。
- `/analysis`：专业分析。
- `/processing`：AI 自动出图。
- `/history`：当前浏览器的本地历史。

### 4.2 专业分析

用户上传 FITS 后，系统输出：

- 真实解析的 FITS 头、HDU 信息、图像尺寸、位深和基础统计。
- Mock 生成的 SNR、FWHM、椭圆率、星点质量等专业指标。
- Mock 生成的分步骤处理方案和推荐参数。

首期不生成 Siril 或其他软件的可执行脚本。

### 4.3 AI 自动出图

用户在提交任务前选择一种风格：

- `realistic`：真实，处理保守，科学真实性优先。
- `balanced`：平衡，兼顾自然观感和细节表现，为默认选项。
- `artistic`：艺术，允许更强的色彩和局部对比，但不得虚构结构。

每个任务只生成一种风格。处理过程完全无人值守，用户可以查看计划、阶段、日志和结果，也可以取消任务。

首期由 Mock Agent 生成演示计划、工具调用、质量评分和演示产物。

### 4.4 历史记录

历史索引只保存在当前浏览器 IndexedDB 中：

- 换设备、无痕模式或清除站点数据后无法恢复。
- 浏览器可能在存储压力下清理 IndexedDB，产品需明确提示本地历史不保证永久保存。
- 服务端文件过期后，本地历史仍保留任务摘要，并显示结果已过期。

### 4.5 非目标

首个里程碑不包含：

- Light、Dark、Flat、Bias 的校准、配准和叠加。
- XISF 支持。
- 真实 SNR、FWHM、星点检测、降噪、反卷积、色彩校准等算法。
- Siril、PixInsight 或其他外部后期软件集成。
- 真实 AI 模型供应商接入。
- 用户账户、跨设备历史、支付和严格配额。
- Redis、Celery、PostgreSQL、对象存储、多 Worker 或多节点计算。
- 从中断步骤恢复任务。

## 5. 输入与输出

### 5.1 输入格式

首期只接受扩展名为 `.fits`、`.fit` 或 `.fts`，结构合法且不超过 500 MB 的 FITS 文件。

系统接受不同合法 FITS 结构，但不承诺全部可处理。上传后扫描所有 HDU：

- 对可归一化为二维单色或三通道 RGB 的图像数据继续处理。
- 对表格、光谱、时间序列、无图像 FITS 和无法支持的高维数据返回明确错误。
- 若存在多个可处理图像 HDU，自动选择像素面积最大的图像。
- 报告记录选中的 HDU 编号、名称、尺寸和选择原因。
- 无可处理图像时不创建任务，不扣每日额度。

### 5.2 输出格式

专业分析输出网页报告。

自动处理输出：

- 一份高质量 TIFF 演示产物。
- 一份 PNG 演示预览图。
- 一份结构化处理清单，记录计划、步骤、参数、工具和 Mock 质量评估。

所有演示产物必须包含清晰的 Mock 标识，避免被误认为真实处理结果。

## 6. 信息架构与用户流程

### 6.1 首页

首页是产品落地页，包含：

- 产品价值与目标用户说明。
- 输入文件要求与能力边界。
- “开始分析”和“自动出图”两个主要入口。
- 上传区。
- 三步流程、隐私说明、24 小时保留规则和当日剩余额度。

首页不展示没有数据来源的用户量、成功率或累计处理量。

### 6.2 分析流程

1. 用户选择 FITS。
2. 浏览器流式上传文件。
3. 服务端验证格式、大小、磁盘空间和 FITS 结构。
4. 服务端扫描 HDU 并选择最大可处理图像。
5. 用户创建分析任务，此时扣减一次额度。
6. 任务进入队列并由串行执行器处理。
7. 页面通过 HTTP 轮询状态和增量日志。
8. 完成后显示真实基础信息、Mock 专业指标和建议。
9. 用户可在文件保留期内复用原文件和基础指标创建自动处理任务。

### 6.3 直接自动处理流程

1. 用户上传并验证 FITS。
2. 用户选择真实、平衡或艺术风格。
3. 用户创建自动处理任务，此时扣减一次额度。
4. 任务排队并由 Agent 执行。
5. 页面展示队列、阶段、计划、工具调用、日志和进度。
6. 完成后展示前后对比和 TIFF、PNG 下载入口。

### 6.4 文件复用

分析完成后，可通过 `source_task_id` 创建自动处理任务：

- 不需要重新上传。
- 复用仍未过期的原文件和真实基础指标。
- 自动处理是独立任务，计入每日 5 次额度。
- 原始分析任务过期、删除或文件已清理后不可复用。

## 7. 页面与交互

### 7.1 视觉方向

沿用原 PRD 的深空暗色界面、暖白正文、红色强调、星场与星云背景。视觉装饰不能降低报告、错误和任务状态的可读性。

### 7.2 任务状态展示

界面统一支持：

- 上传前。
- 上传中。
- 验证中。
- 排队中。
- 处理中。
- 正在取消。
- 已完成。
- 已失败。
- 已取消。
- 已过期。

上传中显示字节进度和取消按钮。刷新页面会中断尚未完成的上传。

处理中默认每 2 秒轮询状态；排队时间较长时可以降低轮询频率。页面恢复后可根据 IndexedDB 中的任务 ID 继续轮询。

完成页显示服务端文件到期时间和倒计时。失败页显示稳定错误码、可读说明、是否可重试和是否扣除额度。

### 7.3 Mock 标识

分析页将真实 FITS 信息与 Mock 专业指标分区展示。

自动处理页在计划、工具日志、预览和下载区域持续显示“演示流程”或“Mock 结果”，不能只在首次进入时提示一次。

### 7.4 浏览器支持

- 支持 Chrome、Edge、Firefox 和 Safari 最新两个大版本。
- 桌面端支持完整上传、分析、处理和下载流程。
- 移动端重点保障首页、报告、历史和任务状态。
- 移动端允许上传，但不承诺 500 MB 文件或切换到后台后持续上传。
- 不支持 IE、内置 WebView 和未列入验收范围的小众浏览器。

### 7.5 国际化

首版只提供简体中文。用户文案、状态名称和错误码不得散落硬编码，应通过可国际化资源组织，为后续英文版本预留。

## 8. 系统架构

### 8.1 总体架构

采用单机轻量架构：

- Next.js Web：页面、交互、IndexedDB 历史。
- FastAPI：上传、任务 API、文件下载和清理管理。
- 应用内任务执行器：消费 SQLite 队列，全局一次只执行一个任务。
- Agent Core：计划、工具注册、调用、观察、质量评估和终止。
- FITS Inspector：真实 FITS 解析和基础统计。
- SQLite：任务、事件和用量。
- 本地磁盘：原文件、预览、中间产物和结果。

Next.js 不使用纯静态导出来承载后端能力。Web 可以独立构建和部署，但上传、任务、轮询和下载均由 Python API 提供。

### 8.2 边界

- Web 不执行图像处理，也不持有服务端文件。
- HTTP 请求不直接等待耗时任务完成。
- Agent 只做编排，不直接执行任意 Python、Shell 或文件操作。
- 工具必须通过白名单注册并声明输入 schema。
- 文件操作限制在当前任务目录内。
- 任务执行器未来可以拆为独立 Worker，而不改变对外 API 和任务模型。

### 8.3 部署

首期在单台服务器部署，当前已知资源为：

- 4 核 CPU。
- 4 GB 内存。
- 无 GPU。
- 约 50 至 100 GB 可用于任务文件的磁盘空间。

该配置只作为当前部署约束，不作为通用性能基线。首期 Mock 流程不承诺真实处理 500 MB 图像。

## 10. API 契约

- `POST /api/uploads`：流式上传并验证 FITS，返回 `upload_id` 和验证摘要。
- `POST /api/tasks/analysis`：从有效 `upload_id` 创建分析任务。
- `POST /api/tasks/process`：从有效 `upload_id` 或 `source_task_id` 创建处理任务。
- `GET /api/tasks/{id}`：获取状态、进度、阶段、错误、摘要和到期时间。
- `GET /api/tasks/{id}/events?after={sequence}`：增量获取任务和 Agent 事件。
- `POST /api/tasks/{id}/cancel`：请求协作式取消。
- `POST /api/tasks/{id}/retry`：在输入文件仍存在时创建新任务。
- `DELETE /api/tasks/{id}`：删除任务和相关文件。
- `GET /api/tasks/{id}/artifacts/{name}`：下载允许的任务产物。
- `GET /api/usage`：获取当前匿名客户端当日已用和剩余额度。

API 返回稳定错误码、中文消息、`retryable` 和 `quota_charged`。未来增加英文时，客户端根据错误码映射本地文案。

## 11. 任务状态机

上传状态：

`uploading -> validating -> ready|invalid`

任务主状态：

`queued -> running -> completed`

可选终态和过渡：

- `queued|running -> cancelling -> cancelled`
- `queued|running|cancelling -> failed`
- `completed|failed|cancelled -> expired`

服务启动时，数据库中残留的 `queued`、`running` 和 `cancelling` 统一转换为：

- `status = failed`
- `error_code = restart_interrupted`
- `retryable = true`

系统不自动恢复排队任务，不从检查点继续。

## 12. 匿名访问与额度

### 12.1 任务访问

首期仅依赖不可预测的任务 ID：

- ID 至少包含 128 位加密随机熵。
- 不提供列举全部匿名任务的接口。
- 查询、取消、删除、重试和下载均实施限流。
- 所有任务和文件最长 24 小时失效。

该方案不是完整对象级授权。独立访问令牌或用户账户是后续安全升级项。

### 12.2 每日额度

- 每个浏览器每天最多成功创建 5 个任务。
- 分析和自动处理共享额度。
- 服务端使用匿名客户端 ID 与 IP 的组合进行容量控制。
- 清除浏览器数据或更换网络可能绕过限制，因此该机制不是严格计费或身份配额。
- 上传、格式验证和不支持数据不扣额度。
- 任务成功创建时扣额度。
- 对已扣额度且因 `restart_interrupted`、资源错误或系统错误失败的任务，`retry` 创建一次不重复扣额度的新任务。
- 已完成、用户取消、输入错误或免费重试已经使用的任务，再次创建任务正常扣额度。

## 13. 文件生命周期

- 未创建任务的有效上传固定保留 1 小时。
- 原文件、中间产物和结果在任务完成、失败或取消后固定保留 24 小时。
- 用户可提前删除任务和全部文件。
- 取消处理中任务采用协作式终止，当前工具在安全取消点停止，然后清理已生成产物。
- 定时清理器删除过期任务文件并将状态更新为 `expired`。
- 清理操作必须幂等，能安全处理进程中断造成的半清理状态。
- 可用磁盘低于安全水位时拒绝新上传。
- 每个任务使用独立目录，文件名由服务端生成，不信任客户端文件名作为路径。

## 14. Agent 设计

### 14.1 生命周期

1. Context：读取任务、FITS 摘要、风格和资源限制。
2. Plan：模型适配器返回结构化计划。
3. Execute：按计划调用白名单工具。
4. Observe：收集工具结果、指标、日志和产物。
5. Evaluate：质量评估器决定完成、失败或有限迭代。
6. Finalize：生成报告和产物清单。

### 14.2 接口

`ModelAdapter`：

- `plan(context)`
- `evaluate(observation)`

首期实现 `DeterministicMockModel`，不调用外部 AI API。

`Tool`：

- 名称和版本。
- 输入与输出 schema。
- 资源声明。
- `execute(context, input)`。
- 可取消点。

`ToolRegistry`：

- 注册白名单工具。
- 根据名称和版本查找工具。
- 禁止模型传入任意模块、函数、Shell 或路径。

`AgentRunner`：

- 验证计划和工具参数。
- 记录步骤、输入摘要、输出摘要、耗时和错误。
- 控制最大步骤和迭代次数。
- 处理取消和超时。

`ArtifactStore`：

- 只允许在当前任务目录内读写。
- 生成名称稳定、类型明确的产物清单。

`QualityEvaluator`：

- 首期返回固定种子生成的确定性 Mock 评分。
- 后续可替换为真实指标和视觉模型。

### 14.3 首期真实能力

- FITS 头与 HDU 列表读取。
- 最大可处理图像 HDU 选择。
- 图像尺寸、位深、数据范围、均值、中位数和标准差。
- 内存映射或分块读取。
- 计划、工具调用、事件日志、取消、超时和产物清单。

### 14.4 首期 Mock 能力

- SNR、FWHM、椭圆率和星点数量。
- 分步骤建议和推荐参数。
- 三种风格的处理计划。
- 拉伸、降噪、反卷积和色彩工具。
- 质量评分、迭代决策和最终成片。

### 14.5 未来模型数据边界

未来接入外部 AI API 时，只允许发送：

- 降采样预览图。
- 结构化基础指标。
- 必要 FITS 元数据。

原始 FITS 和完整中间产物始终留在服务器。

## 15. 资源治理

- 全局同时只运行一个分析或处理任务，其余任务排队。
- 上传采用流式写入，不把完整 500 MB 文件载入内存。
- FITS 读取优先使用内存映射或分块统计，避免不必要的数据复制和类型扩张。
- 分析任务最长 10 分钟。
- 自动处理任务最长 60 分钟。
- 接近进程内存保护阈值时主动终止任务并返回可重试的资源错误。
- 每一步执行前后检查取消标记和磁盘空间。
- 当前 4 GB 内存环境不对 500 MB 文件的真实后期处理做性能承诺。

## 16. 错误处理

错误分为：

- 输入错误：格式、大小、损坏、无可处理图像；不扣额度。
- 资源错误：磁盘不足、内存保护、超时；通常可重试。
- 系统错误：服务重启、工具异常、文件缺失；返回诊断 ID。
- 用户操作：取消、删除和过期；不计为系统故障。

错误响应至少包含：

- `error_code`
- `message`
- `retryable`
- `quota_charged`
- `diagnostic_id`，仅系统错误需要

## 17. 测试策略

### 17.1 单元测试

- 状态转换。
- 每日额度。
- 路径隔离和文件名处理。
- HDU 选择。
- Mock 模型确定性。
- Agent 步骤和迭代限制。
- 清理和过期规则。

### 17.2 API 集成测试

- 流式上传与大小限制。
- 合法、损坏和不支持的 FITS。
- 分析与处理任务创建。

## 18. 里程碑 1 验收标准

1. 用户可以上传结构合法且不超过 500 MB 的 FITS，服务端不把完整文件载入内存。
2. 系统真实列出 HDU，自动选择最大可处理图像，并展示基础元数据与统计。
3. 分析和自动处理任务可以排队、轮询、取消、失败、完成、重试和删除。
4. Mock Agent 生成可复现的计划、工具调用、事件日志、报告和演示产物。
5. 分析完成后可以复用文件创建自动处理任务，无需再次上传。
6. 浏览器 IndexedDB 可以恢复历史；服务端结果过期后仍保留本地摘要。
7. 每日最多成功创建 5 个任务，分析与自动处理共享额度。
8. 服务重启后未完成任务标记为 `restart_interrupted`，页面可解释并提供重试。
9. 超时、空间不足、不支持数据、取消和过期均有稳定错误码和用户提示。
10. Chrome、Edge、Firefox 和 Safari 最新两个大版本完成桌面核心流程；移动端完成浏览流程。

## 19. 后续演进顺序

完成里程碑 1 后，按以下顺序扩展：

1. 接入第一个真实图像分析工具，替换一项 Mock 指标。
2. 接入第一个真实处理工具，形成最小可用真实管线。
3. 将任务执行器拆为独立 Worker。
4. 接入真实模型适配器，并落实外部数据脱敏和审计。
5. 增加 XISF、访问令牌或用户账户。
6. 根据负载再评估 Redis、对象存储、多 Worker 和付费额度。
